# Gemini 2.5 深度搜索与文件分析 SOP (LLM-Optimized)

## 1. 任务定义阶段 (Task Definition)
- **核心逻辑**：将模糊的需求拆解为具体的搜索参数。
- **执行动作**：
  1. 识别任务中的关键实体（如：小米、农民、冯玉宽）。
  2. 识别关键时间节点（如：2025, 2026）。
  3. 识别关键事件性质（如：起诉、投诉、侵权、商标）。

## 2. 自动化搜索阶段 (Automated Search)
- **核心逻辑**：通过生成多种搜索引擎的 URL 并执行，获取多源原始 HTML。
- **执行指令**：
  ```shell
  # 使用 Python 生成 Bing 和 Google 的搜索链接（包含关键词编码）
  python -c "import urllib.parse; print('https://www.bing.com/search?q=' + urllib.parse.quote('关键词1 关键词2'))"
  ```
- **关键规约**：
  - 必须优先检查本地是否存在已下载的相关 HTML 文件。
  - 搜索关键词需包含：主体 + 行为 + 客体 + 时间。

## 3. 文件读取与内容嗅探阶段 (File Reading & Sniffing)
- **核心逻辑**：由于 HTML 文件巨大，严禁全量读取。必须使用“正则采样”或“关键词锚点”。
- **执行指令**：
  - **初筛**：使用 `findstr` (Windows) 或 `grep` (Linux) 快速定位关键词行。
  ```shell
  findstr /i /c:"起诉" /c:"农民" /c:"侵权" *.html
  ```
  - **深度提取**：使用 Python 脚本提取关键词前后的上下文（推荐 50-100 字符）。
  ```python
  import re
  content = open('file.html', 'r', encoding='utf-8', errors='ignore').read()
  # 提取包含 农民+小米+起诉 的上下文片段
  matches = re.findall(r'.{0,100}(?:农民.*小米.*起诉|小米.*农民.*投诉).{0,100}', content)
  ```

## 4. 冲突解决与验证阶段 (Conflict Resolution)
- **核心逻辑**：对抗网络传闻，验证核心事实（时间、人物、具体赔偿金额）。
- **执行规约**：
  1. 区分“传闻金额”与“官方起诉金额”。
  2. 确认事件状态（已起诉、已道歉、舆论发酵中）。

## 5. 最终汇总阶段 (Final Synthesis)
- **输出模板**：
  - **事件定性**：[具体发生了什么]
  - **时间坐标**：[精确到月/周]
  - **关键人物/机构**：[具体姓名及公职/司名]
  - **核心冲突点**：[例如：通用词汇 vs 商标权]
  - **社会反馈**：[舆论走向]

## 6. 异常处理 (Exception Handling)
- **若找不到文件**：重新执行第2步，扩大搜索词范围。
- **若文件过大导致超时/截断**：必须执行第3步的 Python 采样脚本，严禁直接 `read_file`。
- **若 API 报错 (503/429)**：静默等待并重试，或优化搜索策略减少冗余调用。
